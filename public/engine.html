BootHop Engine

Process Flow & Business Logic — Internal Reference Document

This document describes the full end-to-end engine behind BootHop: from a visitor landing on the site to a completed, paid, and verified delivery. It covers every major subsystem: authentication, trip matching, payments, identity verification, and admin oversight.

1. Overview — What BootHop Does

BootHop is a peer-to-peer delivery marketplace. It connects:

  • Booters (Travellers) — people who are already travelling between cities and have spare luggage capacity
  • Hoopers (Senders) — people who want to ship a small item along an existing travel route without using a courier

The platform orchestrates matching, payment escrow, identity verification, and delivery confirmation so that neither party has to trust the other blindly.

2. Authentication Flow

BootHop uses a passwordless OTP email flow — no passwords are ever set or stored.

1
Registration / Login Entry

A user arrives at /register or /login. They enter their email address (and optionally a journey intent on the register page — from/to city, date, role, weight, price).

The journey intent form data is captured in URL query params and pre-filled on the register form.

↓
2
OTP Dispatch — POST /api/auth/send-code

The form submits to /api/auth/send-code. The API:

  1. Normalises the email (lowercase, trimmed)
  2. Generates a 6-character alphanumeric code and its SHA-256 hash
  3. Stores the hash in email_login_codes table with a 15-minute expiry, used = false, and the optional journey_payload (trip details the user filled in)
  4. Sends the code via Resend (transactional email) to the user's inbox

The raw code is never stored — only the hash. This means even a full DB dump cannot reveal valid codes.

↓
3
OTP Verification — POST /api/auth/verify-code

User enters the 6-character code. The API:

  1. Looks up the most recent non-expired, unused code for that email
  2. Checks attempt count (< max_attempts, usually 5) — rate-limiting
  3. Hashes the incoming code and compares to stored hash
  4. On match: marks code as used = true
  5. If a journey_payload exists on the code record: inserts a journey_drafts row with the trip details and sets hasDraft = true
  6. Signs an httpOnly session cookie (JWT via signAppSession) and attaches it to the response

Redirect logic:

  • Has a fresh draft → /journeys?listing=new (banner shown: "Your new listing is now live")
  • No draft, has existing trips → /dashboard
  • No draft, no trips → /intent (onboarding wizard)
↓
4
Session Cookie

All subsequent authenticated requests read the boothop_session httpOnly cookie. API routes parse it manually from the Cookie header and call verifyAppSession(token) to get the user's email. This is used instead of Supabase Auth sessions because Next.js 15 Route Handlers cannot write cookies via the cookies() helper — so the JWT is set directly on the NextResponse.

3. Journey Creation Flow

1
Intent Capture

New users land on /intent. They choose their role (Booter = traveller / Hooper = sender) and fill in route, date, weight, and price. The intent wizard stores data in React state and eventually navigates to /register pre-filled.

↓
2
Journey Draft → Trip

After OTP verification with a journey_payload, a journey_drafts row is created. The match engine (or a background job) converts this draft into a live trips record when it detects a potential match or when the user confirms on the dashboard.

The trips table stores: from_city, to_city, travel_date, weight, price, type (travel/send), email, user_id.

4. Match Engine

The match engine is the core of BootHop. It finds complementary trip pairs and creates a matches record.

1
Match Criteria

A Booter (travel) trip and a Hooper (send) trip are matched when:

  • Route overlap: same origin and destination cities (case-insensitive)
  • Date compatibility: Booter travel date is on or before Hooper required date (within a configurable window)
  • Weight capacity: Booter's available weight ≥ Hooper's item weight
  • Price agreement: price negotiation happens after match notification
↓
2
Match Record Creation

The engine creates a matches row with:

  • booter_email / hooper_email — the two parties
  • booter_user_id / hooper_user_id — linked Supabase Auth user IDs
  • sender_trip_id / traveler_trip_id
  • status: starts as pending
  • agreed_price, hooper_pays, booter_receives — set after price negotiation

Both parties receive Resend notification emails when a match is found.

↓
3
Match Acceptance

Both the Booter and Hooper must accept the match from their dashboards. The match status moves from pending → accepted. Once accepted, the Hooper is directed to pay into escrow.

5. Identity Verification (KYC)

BootHop runs a two-layer identity verification to ensure both parties are who they claim to be.

Layer 1 — Stripe Identity (Document KYC)

Triggered via /api/stripe/identity. The user is directed to Stripe's hosted identity flow where they:

  1. Upload a government-issued ID (passport, driving licence)
  2. Take a selfie for facial comparison

Stripe returns a VerificationSession with extracted fields: first_name, last_name, dob, id_number. On success, the user's profile is marked verified = true.

Stripe Identity webhooks update the profiles table automatically via /api/webhooks/stripe/identity.

Layer 2 — Video KYC (In-Person Liveness)

After a match is accepted, each party must complete a short video verification at /kyc/video?matchId=...:

  1. Intro: User enters their full name (as it appears on their ID)
  2. Recording: 3-second countdown, then up to 20 seconds of video. A script prompt is shown: "My name is [name]"
  3. Photo: After recording, a live selfie snapshot is taken from the camera
  4. Submission: Video (.webm) + photo (.jpg) uploaded to Supabase Storage bucket video-kyc via POST /api/kyc/video-submit

The API stores paths in the matches table (booter_video_kyc_path, hooper_video_kyc_path, etc.) and upserts into the video_kyc table with status = 'pending_review'.

Expiry: Video KYC is valid for 30 days (expires_at = now + 30d). After expiry, the user must re-verify before their next match.

Admin Video KYC Review

Admins review pending submissions in the Admin Control Centre → Video KYC tab:

  1. Admin clicks "Review" on a pending record
  2. Signed storage URLs are generated for the video and photo (1-hour expiry)
  3. Admin watches the video, inspects the photo, checks the submitted name
  4. Admin clicks Approve or Reject
  5. Approve: sets video_kyc.status = 'approved', matches.booter/hooper_video_kyc_status = 'approved', matches.booter/hooper_id_received = true, profiles.id_verified = true
  6. Reject: sets status to 'rejected' — user is prompted to re-record

6. Payment & Escrow Flow

1
Checkout

The Hooper (sender) pays via Stripe Checkout. The amount is agreed_price × (1 + platform_fee%). A pre-payment confirmation page shows the full cost breakdown, Terms & Conditions, and insurance disclosure before the user is taken to Stripe.

The Stripe session is created with payment_intent_data.capture_method = 'automatic' and funds are captured into BootHop's Stripe account.

↓
2
Escrow Hold

On Stripe webhook checkout.session.completed:

  • matches.payment_status is set to 'escrowed'
  • matches.stripe_payment_intent_id is stored
  • Both parties receive a confirmation email via Resend

Funds are held in BootHop's Stripe balance — not released until both parties confirm delivery.

↓
3
Dual Confirmation & Release

After the item is physically delivered:

  1. Booter confirms delivery: sets booter_confirmed_delivery = true
  2. Hooper confirms receipt: sets hooper_confirmed_receipt = true

When both confirmations are present, a backend trigger (or webhook handler) calls stripe.transfers.create to send booter_receives to the Booter's connected Stripe account. The match status moves to 'completed'.

Admin override: Admins can manually release escrow from the Escrow tab in the Admin Control Centre — this bypasses dual confirmation and should only be used in exceptional circumstances.

7. Fee Structure

Party What they pay / receive Notes
Hooper (sender) agreed_price × 1.03 (+ Stripe fees) 3% platform fee on top of agreed price
Booter (traveller) agreed_price × 0.97 Receives 97% of agreed price after platform cut
BootHop ~6% of agreed_price 3% charged to sender + 3% retained from traveller payout

11. Technology Stack

  • Frontend: Next.js 15 (App Router), React 19, TypeScript, Tailwind CSS 4
  • Backend / API: Next.js Route Handlers (serverless functions on Vercel)
  • Database: Supabase (PostgreSQL) — auth, profiles, trips, matches, video_kyc
  • Storage: Supabase Storage — video-kyc bucket for video/photo files
  • Payments: Stripe Checkout + Stripe Connect for payouts + Stripe Identity for KYC
  • Email: Resend — transactional emails (OTP, match, payment, KYC)
  • Hosting: Vercel (PWA-enabled, service worker)
  • Session: Custom httpOnly JWT cookie (boothop_session), signed with jsonwebtoken

12. End-to-End Happy Path

  1. Booter visits /register, fills in London → Paris, 5 June, 10 kg available, £30 price → submits email
  2. Receives OTP → enters code → session cookie set → redirected to /journeys?listing=new with success banner
  3. Hooper visits /journeys, sees London → Paris trip, clicks "I'm interested!" → enters own details → receives OTP → logs in
  4. Match engine finds pair → creates matches record → both receive notification emails
  5. Both accept the match from their dashboards → status → accepted
  6. Stripe Identity KYC triggered for both → documents verified by Stripe
  7. Video KYC: both record short video + selfie → admin reviews and approves → id_received = true
  8. Hooper pays £30.90 via Stripe Checkout → funds escrowed in BootHop's Stripe account
  9. Travel date arrives → Booter carries item → hands over to Hooper
  10. Booter confirms delivery + Hooper confirms receipt (both via dashboard)
  11. Stripe transfer: Booter receives £29.10 → match marked completed
  12. Both receive completion confirmation emails
Note on current state: Some steps (e.g. automatic trip draft → live trip conversion, dual-confirmation Stripe payout trigger) are partially manual / admin-assisted in the current implementation. These are identified as next engineering milestones.
