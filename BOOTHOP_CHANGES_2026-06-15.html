Africa-outbound matches — 7-day expiry added to cron Critical
src/app/api/cron/expire-matches/route.ts

awaiting_authorisation status was never included in the expiry cron. If admin never approved/rejected, the match and both parties' trips would be locked permanently — sender couldn't rebook, traveller couldn't take new matches.

🔴 Was it needed? Yes — without this, every Africa-outbound match that admin ignored would lock two users' trips forever. No recovery path existed.
Critical fix
3
Cron schedule — daily 03:00 → every 6 hours
vercel.json

The expire-matches cron ran once per day at 03:00 UTC. The system has a 4-hour timeout for matched status. A match that expired at 03:01 would wait nearly 24 hours before being cancelled and trips restored.

Fix: Changed to 0 */6 * * * — runs at 00:00, 06:00, 12:00, 18:00. Maximum wait for any timeout is now 6 hours.

✅ Was it needed? Yes — the 4-hour and 12-hour thresholds existed precisely to enforce tight lifecycle windows. Running the cron daily made those thresholds meaningless.
Needed
4
Barcode scan — status guard added
src/app/api/tracking/scan-sender-barcode/route.ts

No lifecycle check existed on the barcode scan endpoint. Senders could request the traveller's location during matched, agreed, or kyc_pending — before either party had committed to or paid for the delivery.

Fix: Added guard — only processes scan when match status is active or delivery_confirmed.

✅ Was it needed? Yes — location data is sensitive. Exposing traveller location before payment and KYC are complete is both a privacy risk and a logical error in the delivery lifecycle.
Needed

sign-in pages  OTP + auth fixes (all 3 sign-in pages)

14
OTP input completely broken — regex stripped the letter Critical
sign-in/page.tsx · priority-sign-in/page.tsx · carrier-sign-in/page.tsx

The OTP format (from src/lib/auth/code.ts) is 4 digits + 1 letter = 5 chars (e.g. 4827A). All three sign-in pages used .replace(/\D/g, '') which strips ALL non-digits — the letter was removed before the code could be submitted. Button was also disabled={otp.length < 6} but the max possible length after stripping was 4. Login was impossible.

Fix: Changed to .replace(/[^A-Z0-9]/gi, '').toUpperCase(). Fixed maxLength 6→5, disabled threshold 6→5, placeholder 6 blanks→5, description "6-digit"→"5-character". Added uppercase CSS class. Carrier: also fixed inputMode="numeric"→"text" (numeric keyboard on mobile has no letter key).

🔴 Was it needed? Yes — this is the most critical fix of the session. Nobody could sign into any business account (Express, Priority, or Carrier) through the OTP flow. The login was 100% broken.
Critical fix
15
Priority sign-in — always routed to priority portal regardless of account type
src/app/business/priority-sign-in/page.tsx

The new priority sign-in page originally hard-coded router.push('/business/portal/priority') after a successful OTP. An Express client who accidentally landed on the priority sign-in page would be pushed to the priority portal, which would then bounce them back to the Express portal. Two unnecessary redirects, with a brief flash of the wrong UI.

Fix: After OTP verify and on skipOtp, calls /api/business/auth/me and routes based on partner_status. Express clients go to Express portal directly.

✅ Was it needed? Yes — introduced by us when creating the page. Caught and fixed before committing. Correct routing matters: the priority portal is restricted and would have caused a visible redirect bounce for non-priority users.
Needed
16
Carrier sign-in — mobile keyboard locks out letter entry
src/app/business/carrier-sign-in/page.tsx

The OTP input used inputMode="numeric". On iOS and Android, this forces the numeric keyboard — which has no letter keys. Carrier users on mobile could type the 4 digits but had no way to enter the required letter. Even if they somehow did (e.g. switching keyboards), the old regex would strip it. Combined with the OTP bug (#14), carriers on mobile had zero path to login.

Fix: Changed to inputMode="text". Standard keyboard shows on mobile, allowing the full alphanumeric code.

🔴 Was it needed? Yes — carriers are the most mobile-heavy user type (they're on the road). Making the login keyboard physically unable to enter the required character completely blocks their most common device type.
Critical fix

Debatable Changes — Honest Assessment

These 2 changes are improvements, not fixes. The app would have worked without them. That said, both are low risk and the right call — just worth noting they weren't "must-do" items.
17
Remove client-side personal email check (Express + Carrier sign-in)
src/app/business/sign-in/page.tsx · carrier-sign-in/page.tsx

Both pages had a client-side PERSONAL_DOMAINS check using email.includes('gmail.') substring matching. The API (send-otp) already validates this using an accurate domain-split check and returns the same error message.

The client check was redundant. The substring approach (e.g. checking for 'gmail.') is also slightly less accurate than the API's domain.split('@')[1] approach.

Before: Instant client-side error. After: One network round-trip (~150ms) then same error from API.

🟡 Was it needed? Debatable. The API already handled this correctly. The only real gain is removing a redundant code path and a mildly inaccurate substring check. The downside: 150ms slower error feedback. Low risk either way.
Debatable
18
Auth check on mount — redirect if already signed in (all 3 pages)
sign-in/page.tsx · priority-sign-in/page.tsx · carrier-sign-in/page.tsx

Added a useEffect on mount that calls /api/business/auth/me. If the user already has a valid biz_session cookie, they are redirected straight to their portal instead of seeing the sign-in form.

What this prevents: A logged-in user navigating to /business/sign-in and seeing a sign-in form.

Trade-off: Every page load of any business sign-in page now makes a /me API call. For unauthenticated users (the vast majority of sign-in page visitors) this returns 401 in ~100ms and setChecking(false) is called. A brief spinner is shown.

🟡 Was it needed? Debatable. A logged-in user landing on the sign-in form isn't broken — the form still works, they'd just enter their email and be redirected. The auth-on-mount check is a UX polish. Risk: if /me is slow, the spinner lingers. The catch block handles failure gracefully.
Debatable

Overall Verdict

4 changes were outright login/payment blockers: The OTP input stripping the letter (all 3 business sign-ins), the mobile keyboard locking carriers out, the messages inbox querying dead tables, and the payment success page 404. Any one of these would have generated immediate critical support tickets on launch day.
12 changes fixed genuine bugs that would have surfaced within the first week: broken discount badges, no payment emails, locked trips, ghost jobs, wrong portal branding for priority clients.
2 changes are quality improvements (auth redirect on mount, remove redundant email check). They improve UX and code quality but would not have caused support tickets. Both are low-risk and the right call — just honest to label them as polish rather than fixes.

Still Open — Fix 5 (OTP Expired UX)

Intentionally deferred. When a user submits an expired OTP, they currently see the error "Verification code has expired. Please request a new one." and must manually click "Use a different email" to reset. The better approach — to be discussed separately — is to make that error message itself actionable (e.g. a link/button that resets to the email stage) rather than auto-resetting based on string matching in the error message.
